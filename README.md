# java-blackjack

## 블랙잭 미션 사전 학습 저장소

- 입출력(Console/Scanner/Random)을 제외한 작은 순수 계산 규칙 하나만 선택한다.

- 예시 (이 중 하나만 선택):

| 규칙           | 설명                        | 적정 크기    |
|--------------|---------------------------|----------|
| 카드 점수 계산     | 숫자 카드는 숫자 그대로, J/Q/K는 10점 | 메서드 1-2개 |
| Ace의 1/11 처리 | 합계에 따라 1 또는 11로 계산        | 메서드 1-2개 |
| 버스트 판정       | 합계가 21 초과인지 판단            | 메서드 1개   |
| 카드 합계        | 여러 카드의 점수 합산              | 메서드 1-2개 |

> 적정 크기: 메서드 1-3개, 클래스 1개 정도.
> 30분 안에 구현할 수 있는 크기가 적당하다.
> 너무 크면 실험의 초점이 흐려진다.

## 실험1: main 기반 검증의 한계

### 해볼 것

1. 선택한 규칙을 main 메서드에서 구현
2. 테스트 케이스 3개로 실행해본다
3. 케이스를 6개로 늘려본다

### 관찰할 것

- 케이스가 늘어나니 어디를 수정해야 했는가?
    - Card 클래스에서 수정이 필요했다. 카드 생성시 값을 관리하도록 수정해야했다.
- 수정하면서 놓친 것은 없는가?
    - 첫 번째 수정에서 'A', 'J', 'Q', 'K' 외의 다른 알파벳은 받지 않도록 수정했다.
    - 잘 캐치했다고 생각 하지만 다른 놓친 사항이 있는지 정확히 확인할 수 없다.
- 이전 케이스가 여전히 동작하는지 어떻게 확인했는가?
    - 직접 재실행, 입력, 확인해야 했다.

### 기록하기

형식: 각 관찰은 사실 1개 + 문제 1개

```
- 사실: 
main에서 Card값을 빼오고, 알파벳인지 아닌지 판단하고, 
'A'라면 1을, 아니라면 10을 반환하는 로직을 짰었다.
테스트 케이스를 늘리면서 main이 Card 보다 Card의 field(value)를 더 잘 알게되는 현상이 나타났다.

- 문제: 
value가 가져야할 정책('A'는 1, 'J', 'Q', 'K'는 10)을 main이 관리하게 되었다.
때문에 정책을 수정하고 싶을 때 main을 수정하게 되었고, 다시 main 레벨(입출력)에서 테스트 해야했다.
```

## 실험2: 테스트 코드 작성

### 해볼 것

1. 같은 로직을 JUnit 테스트로 작성 (최소 3개)
2. 경계 케이스나 예외 케이스 1개 추가
3. 테스트 작성이 어려운 부분을 표시

### 관찰할 것

- 테스트를 작성하기 어려운 코드가 있는가?
    - 크게 없었다. 하지만 main을 테스트 하려고 했다면 어려움이 있었을 것이다.
- 어렵다면, 무엇 때문에 어려운가?
    - 사용자의 입력이나 출력을 테스트하는 것은 쉽지 않기 때문이다.
    - 개인적인 생각으로는 입력이나 출력은 테스트할 필요가 없다고 생각한다.
- 테스트를 쉽게 하려면 코드를 어떻게 바꾸고 싶은가?
    - 현재 입력과 출력은 테스트할 필요가 없도록 되어있다.
    - main은 입출력을 담당하고, Card는 생성 정책과 반환 로직을 가지고 있기 때문이다.
    - 하지만 조금 더 보완할 점을 찾는다면, 현재 Card 생성 정책이 코드에 뿌려져있는 것이다.
    - 코드에 정책이 흩뿌려져있다면 추가, 삭제, 변경에 용이하지 않다고 생각한다.
    - 따라서 정책(A, J, Q, K 만 허용 등)을 관리하는 클래스나 인터페이스를 두어 Card에 적용하도록 변경해보고 싶다.

### 기록하기

```
관찰 기록에 추가:

- 테스트가 어려웠던 코드 1곳:
현재 코드에서 큰 어려움은 없었다. 굳이 생각해보자면 main 메서드(입출력)를 테스트하려고 했던 것이다.
한가지 의문은 "테스트 코드에는 꼭 해피 케이스가 있어야하나?"이다. 
있어야 한다면 얼마나 있어야하는지, 예외 케이스와의 비율은 어느정도면 좋을지 고민이다.

- 바꾸고 싶은 부분과 이유:
현재는 코드에 정책이 흩뿌려져있다.
예를 들어, 생성자의 인자를 validate할 때 A, J, Q, K를 검사하고,
value를 score로 내보내는 메서드에서 A, J, Q, K를 확인한다.

만약 정책이 바뀐다면, 여러 메서드에서 그것을 수정해야한다.
따라서 이 부분을 카드 정책을 관리하는 분리하여 수정하고 싶다.

```
